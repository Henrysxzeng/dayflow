## 二、技术选型与理由

### 前端：微信小程序（原生）

| 选项 | 理由 |
|------|------|
| **原生 WXML/JS** ✓ | MVP 阶段最快，无额外依赖，调试简单 |
| Taro / UniApp | 跨平台，但 MVP 阶段过度设计 |

**开发工具：** 微信开发者工具（官方 IDE）

---

### 后端：腾讯云·云开发（CloudBase）

这是 MVP 阶段的最优解，理由：

| 能力 | 说明 |
|------|------|
| 免运维 | 无需购买服务器、配置 Nginx、管理部署 |
| 原生微信集成 | 用户登录、订阅消息、支付均有官方 SDK |
| 免费额度 | 1GB 数据库 + 5GB 存储 + 100 万次云函数调用/月，MVP 完全够用 |
| 定时触发器 | 原生支持 Cron，直接实现"每天 07:50 生成计划并推送" |
| 冷启动后可迁移 | 用户量大后可无缝迁移至独立 Python/Node 服务 |

**云函数语言：** Node.js 18（云开发官方支持最好）

---

### 数据库：云开发 CloudDB（MongoDB）

云开发内置 MongoDB，无需额外配置，适合 MVP。

**后期扩展：** 用户量 > 1 万后，考虑迁移至腾讯云 PostgreSQL（关系型查询更高效）

---

### AI 服务：DeepSeek API（首选）

| 对比项 | DeepSeek | 通义千问 | GPT-4o |
|--------|---------|---------|--------|
| 中文理解 | 极强 | 强 | 强 |
| 响应速度 | 快 | 中 | 中 |
| 价格（百万 token） | ~¥1 | ~¥4 | ~¥35 |
| 国内访问 | 直连 | 直连 | 需代理 |
| **结论** | **首选** | 备选 | 不用 |

**每次 AI 调用的 token 估算：**
- 系统 prompt + 用户任务列表：约 800 token
- AI 返回计划：约 400 token
- 合计：约 1200 token/次 ≈ ¥0.0012/用户/天
- 1000 用户 × 30 天 ≈ **¥36/月**，极低成本

## 三、数据库设计

### 集合（Collections）

#### 3.1 `users` 用户表

```javascript
{
  _id: "openid_xxxxx",          // 微信 openid，作为主键
  nickname: "张三",
  avatar_url: "https://...",
  settings: {
    wake_time: "08:00",         // 推送时间，默认 08:00
    default_daily_hours: 4,     // 默认可用时长（首次未填时使用）
    ai_tone: "friendly"         // friendly | strict | humorous
  },
  streak: {
    current: 12,                // 当前连续天数
    longest: 25,                // 历史最长
    last_active_date: "2026-05-16",
    jokers_remaining: 1         // 免死金牌剩余次数
  },
  created_at: Date,
  updated_at: Date
}
```

#### 3.2 `tasks` 任务表

```javascript
{
  _id: "task_id_auto",
  user_id: "openid_xxxxx",
  title: "完成 Q2 报告",
  description: "包含数据分析和可视化部分",
  deadline: Date,               // 截止日期
  estimated_minutes: 90,        // 预计用时（分钟）
  difficulty: 3,                // 1-5，用户填写
  urgency: 2,                   // 1-4（AI 或用户评定）
  importance: 4,                // 1-4（用户填写）
  quadrant: "Q1",               // Q1重要紧急 Q2重要不紧急 Q3紧急不重要 Q4都不重要
  is_fragment: false,           // true = 碎片任务（≤10分钟），进随手清空列表
  status: "pending",            // pending | in_plan | completed | deferred
  fail_history: [               // 未完成记录
    {
      date: "2026-05-15",
      reason: "too_hard"        // too_hard | no_time | dont_want
    }
  ],
  fail_count: 1,
  parent_task_id: null,         // 非空时表示是某任务拆解出的子任务
  created_at: Date,
  completed_at: null
}
```

#### 3.3 `daily_plans` 每日计划表

```javascript
{
  _id: "plan_id_auto",
  user_id: "openid_xxxxx",
  plan_date: "2026-05-17",
  available_hours: 3,           // 用户当天填写的可用时长
  hours_source: "user_input",   // user_input | default

  next_day_auth: true           // 是否已获得次日推送授权
}
```

## 四、AI 调用设计

### 4.1 Prompt 设计（任务排期）

```
[System Prompt]
你是用户的私人效率助手，名叫 Flow。你的任务是根据用户今天的可用时间和任务清单，
帮他选出今天最该做的 3-5 件事，并给出具体的时间安排建议。

规则：
1. 优先选 Q1（重要且紧急）的任务
2. 预计总用时不能超过用户今日可用时间的 85%（留 15% 缓冲）
3. 如果某任务历史上连续标记"太难了"超过 2 次，自动拆解为更小的子任务
4. 碎片任务（≤10 分钟）不进入主计划，另列"随手清空"清单
5. 语气温暖、简洁，像朋友说话，不要说教
6. 返回严格的 JSON 格式，不要有多余文字

[User Prompt]
今天日期：2026-05-17（星期日）
今日可用时间：3 小时

待办任务清单：
[
  {"id": "t1", "title": "完成 Q2 报告", "deadline": "2026-05-19", "estimated_minutes": 90, "quadrant": "Q1", "fail_count": 0},
  {"id": "t2", "title": "回复客户邮件", "deadline": "2026-05-17", "estimated_minutes": 20, "quadrant": "Q1", "fail_count": 0},
  {"id": "t3", "title": "读《原子习惯》第3章", "deadline": "2026-05-31", "estimated_minutes": 30, "quadrant": "Q2", "fail_count": 0},
  {"id": "t4", "title": "买生活用品", "deadline": "2026-05-18", "estimated_minutes": 8, "quadrant": "Q3", "fail_count": 0},
  ...
]
```

```

### 4.3 任务拆解 Prompt（触发条件：fail_count ≥ 2 且原因为 too_hard）

```
任务"[title]"用户已连续标记"太难了" 2 次，预计用时 [X] 分钟。
请将其拆解为 3-5 个更小的子任务，每个子任务预计用时不超过 20 分钟。
返回 JSON 数组，格式同任务结构。

## 五、核心业务流程

### 5.1 晨间推送流程（每日 07:50 定时触发）

```
定时触发器 07:50
    │
    ├─ 查询所有昨日已授权推送的用户
    │
    ├─ 对每个用户：
    │     ├─ 读取其待办任务池
    │     ├─ 调用 DeepSeek API 生成初步计划（用默认时长）
    │     └─ 写入 daily_plans 表（hours_source: "default"）
    │
    └─ 08:00 调用微信订阅消息 API 批量推送
         推送内容：
         "Flow 早！今天有 X 件要事在等你。
          今天能挤出几个小时？回复后 Flow 给你最终版计划 ✓"
```

### 5.2 用户回复可用时间流程

```
用户在小程序点击"今天有 N 小时"
    │
    ├─ 更新 daily_plans.available_hours
    ├─ 重新调用 AI 生成精确计划
    └─ 返回最终计划展示页
         （如 30 分钟内未回复 → 默认时长计划自动生效）
```

### 5.3 任务完成 + 推送授权流程

```
用户勾选今日最后一个任务
    │
    ├─ 更新 task.status = "completed"
    ├─ 更新 daily_logs
    ├─ 检查 streak（+1 或断掉）
    ├─ 展示完成反馈（有温度的一句话）
    │
    └─ 弹出订阅消息授权弹窗
         "明天的计划，Flow 帮你提前想好？"
         [授权明天推送]  [不用了]
              │
         调用 wx.requestSubscribeMessage()
         写入 push_auth 表
```

### 5.4 未完成任务处理流程（每天 23:00 自动执行）

```
定时触发器 23:00
    │
    ├─ 查询当天计划中 status = "in_plan" 的任务
    │
    ├─ 对每个未完成任务：
    │     ├─ 推送轻量问询（如用户当天有打开过 App）：
    │     │   "Q2 报告没做完，标一下原因？"
    │     │   [太难了] [没时间] [不想做]
    │     │
    │     └─ 24 小时内无操作 → 默认标记"没时间"，直接结转
    │
    └─ 根据 fail_history 更新任务状态：
          too_hard × 2 → 触发拆解流程（下次排期前执行）
          dont_want × 2 → 降至 Q3，本周末重新评估
```

## 六、微信订阅消息模板设计

申请微信订阅消息模板，建议申请以下类型：

| 模板类型 | 用途 | 触发场景 |
|---------|------|---------|
| 待办提醒通知 | 晨间计划推送 | 每日 08:00 |
| 任务完成提醒 | 今日任务总结 | 用户完成所有任务 |
| 服务进度通知 | 计划重新生成完成 | 用户点击重新生成后 |

**晨间推送模板字段示例：**
```
标题：Flow 早！今天的计划准备好了
今日任务数：3 件
最重要的一件：完成 Q2 报告（90 分钟）
提示语：今天搞定这 3 件，就是赢的一天
跳转页面：/pages/today/today
```

## 十、技术风险与应对

| 风险 | 概率 | 应对方案 |
|------|------|---------|
| 微信订阅消息模板审核不通过 | 中 | 提前申请多个模板备用；初期用服务通知替代 |
| DeepSeek API 响应超时 | 低 | 设置 10 秒超时 + 重试机制；备选通义千问 |
| 云开发免费额度超限 | 低（MVP 阶段） | 监控用量，超限前升级套餐 |
| 用户不给推送授权 | 中 | 优化授权时机（勾选完成时），文案强调价值 |
